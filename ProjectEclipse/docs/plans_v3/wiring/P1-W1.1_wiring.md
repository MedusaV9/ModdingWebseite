# P1-W1.1 — Vanilla pipeline core wiring

Owner: `worldgen/vanilla/*` + `worldgen/DiscChunkGenerator.java`. Integrator / sibling
workers apply these hooks; **do not** edit `EclipseMod.java`, `EclipseCommands.java`,
`network/EclipsePayloads.java` or lang JSONs from this package.

## Required sibling: `FrozenParams` (W1.9)

`DiscGenPipeline.fixedSeed()` calls
`dev.projecteclipse.eclipse.worldgen.FrozenParams.mapSeed()` per §3.10. **W1.9 must land
`FrozenParams.java`** before this package compiles. Expected signature:

```java
public static long mapSeed();
```

It must be callable on **worldgen worker threads** (volatile context) and must return the
frozen per-save map seed (default `DiscMapData.ECLIPSE_SEED`) from `ServerAboutToStartEvent`
on — every carve/decorate/seed phase of the pipeline seeds itself from it.

## Required sibling: `VanillaLandmarks` (W1.6)

`DiscChunkGenerator.findNearestMapStructure` reads the locate table from
`dev.projecteclipse.eclipse.worldgen.structure.VanillaLandmarks.locateSites()` per §3.10.
Expected signature:

```java
public static java.util.Map<ResourceLocation, String> locateSites();
```

(vanilla structure id → landmark id, e.g. `minecraft:desert_pyramid` →
`"eclipse:desert_temple"`). To not regress `/locate`, the table must contain at least the
five entries the generator used to hardcode: `desert_pyramid → eclipse:desert_temple`,
`jungle_pyramid → eclipse:jungle_temple`, `village_plains → eclipse:village_plains`,
`stronghold → eclipse:stronghold_emergence`, `fortress → eclipse:fortress_core`. Must be
thread-safe for read (locate runs on the server thread; an immutable `Map.of`/frozen map is
fine).

## `GrowthPacing.reload` must push the feature deny-list (W1.5)

`worldgen_tuning.json → features.deny[]` (list of placed-feature ids, e.g.
`"minecraft:lake_lava_surface"`) belongs to W1.5's loader. On every `reload(Path)` call:

```java
BiomeFeatureFilter.setConfigDeny(parsedDenyIds); // Collection<ResourceLocation>
```

The hardcoded vanilla mineral-ore deny-list is always active on top of this. **Freeze
semantics**: the effective deny-list freezes at the first `settingsFor` call (vanilla
memoises its feature-step order from it — a later change would desynchronise feature
indices). Push the config **before any chunk decorates** (i.e. from the initial
`EclipseConfig.reload()` during server start, before the first FEATURES chunk). A later
push with different content logs a warning and needs a server restart to apply — surface
that in the `/eclipse reload` feedback if desired.

## `DiscGenPipeline.runOnLiveChunk(ServerLevel, LevelChunk)` (W1.5 ring sweep, W1.6 site prep)

Contract for callers:

- **Server thread only** (decoration writes through `ServerLevel.setBlock`, seeding adds
  entities).
- Call **after** the chunk's base terrain has been rewritten from the terrain function at
  the new committed stage (`WorldStageAccess.setStage` must already report the new stage —
  `HullRepair` and the carver engine evaluate columns at the committed stage), and
  **before** the sweep's own final heightmap re-prime / relight / resend. The pipeline
  primes the four decoration heightmaps itself before decorating (same set as vanilla's
  FEATURES task); lighting/resend stays with the existing sweep machinery.
- Chunks with **no solid sections are skipped** (provably a no-op on the chunkgen path, and
  it protects against replaying before the base rewrite).
- The GROW rewrite must clear `bottomY..maxY` outside the new column solid span (as ERASE
  already does) before calling — otherwise old decoration remains and duplicates.
- **Replay order**: replay rewritten chunks in a deterministic order (e.g. ascending
  chunk x, then z). Features may write across chunk borders; a deterministic order keeps
  border blocks byte-identical between two saves running the same sweep.
- **Protected zones**: the seam carries no structure-box list. If a rewritten chunk
  intersects a structure protection box, the caller owns the guard (skip the chunk's
  replay or re-assert the protected blocks afterwards, per the existing RingGrowth
  protection mechanism).
- Neighbour chunks (±1) should be loaded; decoration reads the 3×3 biome neighbourhood
  through `ServerLevel.getChunk` (would sync-load otherwise).

## `DiscGenPipeline.registerExtraDecor(DiscProfile, ExtraDecor)` (W1.7)

Register at `@EventBusSubscriber` setup time (thread-safe). Decors run **after**
`HullRepair.afterDecoration` on both the chunkgen and live paths, so intentional hanging /
underside dressing is never wiped by the hull guard. The `WorldGenLevel` handed to
`decorate` reports the **frozen map seed** from `getSeed()` — derive any per-chunk random
from it for save-independent output. On the chunkgen path writes must respect
`WorldGenRegion`'s ±1-chunk discipline (`ensureCanWrite`).
