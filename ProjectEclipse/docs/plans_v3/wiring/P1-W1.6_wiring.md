## 2. Frozen trigger API for P2-W7 (rift) and P5 (dev commands)

```java
// PENDING phase: fired the moment a site is enqueued (stage terrain committed).
S2CStructureRiftPayload { String siteId, String structureId, BlockPos anchor, int footprint }
// footprint = expected XZ extent in blocks; plan R11 rift width ≈ footprint * 1.7.

// server side:
StructurePendingRegistry.trigger(String siteId) -> boolean   // place NOW (rift landed)
StructurePendingRegistry.addListener((level, site, phase) -> ...) // Phase.PENDING / Phase.PLACED
StructurePendingRegistry.pending()                           // List<PendingSite> snapshot
StructurePendingRegistry.wasPlaced(String siteId)            // dedupe / introspection
```

- Auto-fallback: a pending site places itself after `structure_phase.auto_delay_ticks`
  (default 100) even if no client/P2 ever triggers — never lost, restart-safe
  (rows persist in `world/<save>/eclipse/pending_structures.json`).
- At most one site places per `structure_phase.place_interval_ticks` (default 20) —
  a stage with mansion + ancient city + mineshafts staggers its paste cost.
- **P2 note**: site `eclipse:stronghold_emergence` starts the quake/fissure sequence on
  trigger (it IS an animation). P2 should either trigger it immediately or suppress its
  own rift visual for that one structureId.

## 3. W1.5 seam — `EclipseWorldgenState` migration

The plan stores pending rows in `EclipseWorldgenState` (SavedData). That class is not in
the tree yet, so `StructurePendingRegistry` isolates storage behind two private methods
(`persist()` / `loadFromDisk()`) writing `world/<save>/eclipse/pending_structures.json`.
When W1.5 lands: replace those two methods with the SavedData accessors; the JSON file
can be dropped (rows are small, no migration needed for dev saves).

## 4. P6 seam — dungeon spawner mob ids (`config/eclipse/dungeons.json`)

Auto-written on first run, live-reloadable via `/eclipse reload` (`ReloadHooks` key
`"dungeons"`; placed spawners re-target in place):

```json
{ "structure_phase": { "auto_delay_ticks": 100, "place_interval_ticks": 20 },
  "collapsed_vault": { "spawners": ["minecraft:zombie", "minecraft:skeleton"] },
  "umbral_warrens":  { "spawners": ["minecraft:cave_spider", "minecraft:zombie"] },
  "monster_room":    { "spawners": ["minecraft:zombie", "minecraft:skeleton", "minecraft:spider"] } }
```

P6 drops `eclipse:*` mob ids into the arrays (index rotates per spawner slot). Unknown /
not-yet-merged ids fall back to the vanilla defaults above with one warning each — a
typo can never break dungeon placement. API: `DungeonSpawners.spawnerTypeFor(id, index)`,
`DungeonSpawners.applyTo(level, pos, id, index)`.

## 5. Locate table (`VanillaLandmarks.locateSites()`, compile seam W1.1)

`desert_pyramid, jungle_pyramid, village_plains, stronghold, fortress` (regression set)
plus `mansion, pillager_outpost, trial_chambers, ancient_city, mineshaft,
bastion_remnant, end_city` → `eclipse:*` landmark ids. Notes:

- `minecraft:mineshaft` maps to landmark id `eclipse:mineshaft`, but mineshaft sites are
  hash-driven (`UndergroundSites`), NOT authored landmarks — `/locate` resolves only ids
  present in `disc_map.json`/`DiscMapDefaults`. If mineshaft locate matters, W1.4 should
  add `eclipse:mineshaft` landmark rows at the deterministic positions (or the generator
  learns a second lookup); positions are stable per save
  (`UndergroundSites.sitesFor(OVERWORLD, stage, 0)`).
- `bastion_remnant` / `end_city` rows are pre-seeded for W1.7/W1.8 (they place through
  `VanillaLandmarks.placeVanilla(...)` + the pending registry and add their landmarks).

## 7. Misc integration notes

- **Seed salts**: `UndergroundSites` burns salts 61–64 of the `ECLIPSE_SEED` hash family
  (mineshaft/vault/warrens/monster-room rolls) — add to the salt registry in
  `P1-W1.2_wiring.md` on merge.
- **Deep-dark tie**: plan D6 anchors the ancient city at y ≈ −40, but W1.3's
  `CaveBiomeMap` starts deep dark below y = −96 — the city envelope sits above the
  deep-dark band. Knob decision for the orchestrator: lower the city anchor
  (`StructureStamper.ANCIENT_CITY_Y`) or raise `CaveBiomeMap.DEEP_DARK_MAX_Y`.
  The Umbral Warrens ARE inside the band (y = −106, r = 112 off the mountain).
- **FogStormSites (W1.9)**: can now migrate to the registry exactly as its comment
  block plans (`enqueue` on stage complete, `materializeSite` from a registered placer;
  `registerPlacer("eclipse:fog_storm", ...)` before `ServerStartedEvent`).
- **Protected zones**: landmark-backed sites are covered by RingGrowthService's existing
  measured-extent path automatically. `UndergroundSites` rows are not landmarks — they
  are deep underground and only re-rolled on erase+regrow (registry clears them via
  `clearPlacedAbove`), so no RingGrowthService edit is needed (per plan).

  en_us + de_de). Nothing reads them yet — merge whenever.
